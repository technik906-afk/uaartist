## ✅ Этап 1. Модель данных под мультикатегорийный магазин

**Цель:** спроектировать данные так, чтобы косметичка была частным случаем товара, а не хардкодом. Самый важный этап — ошибки здесь дороже всего исправлять.

**Задачи:**
- **Category** — с полем `parent` (self-FK) на случай вложенности (Косметички, Сумки, Аксессуары…).
- **Product** — `name`, `slug` (SEO-URL), `description`, `category` (FK), `is_active`, `created/updated`.
- **ProductImage** — отдельная модель (много изображений на товар), порядок сортировки, «главное» изображение.
- **Варианты и опции** — обобщение того, что зашито в `constructor.js`:
  - **Attribute / AttributeValue** — справочники (Размер: Мини/Стандарт/Макси; Цвет: Беж/Шалфей…).
  - **ProductVariant** — конкретная комбинация с `SKU`, ценой и остатком на складе. В корзину добавляется вариант, а не товар.
- **Складские остатки.** Поле `stock` на варианте + поведение при нуле (скрывать кнопку / «нет в наличии»).
- **SEO-поля** на товаре/категории: `meta_title`, `meta_description`.
- **Развилка по конструктору** (влияет на Этап 3): custom-косметичка — это не заранее заведённый вариант. Решение: `OrderItem` умеет ссылаться либо на `ProductVariant`, либо хранить кастомную конфигурацию (JSON). Генерировать варианты под все комбинации — не рекомендуется.

**Ключевые решения:** цена на товаре или на варианте; как моделируется кастомный товар из конструктора; нужна ли иерархия категорий сразу.

**Готово, когда:** через Django admin можно завести категорию, товар с несколькими фото, вариантами, ценами и остатками; данные покрывают и «готовые» товары, и кастомную позицию конструктора.

## ✅ Этап 3. Backend: корзина и заказы

**Цель:** довести заказ надёжно до бэкенда (сейчас он в localStorage и улетает в Google Apps Script → Telegram, минуя БД).

**Задачи:**
- **Развилка по корзине** (влияет на Этап 4): клиентская (localStorage) vs серверная (сессия/пользователь). Рекомендация для MVP — клиентская, но с валидацией цен и остатков на бэкенде при оформлении. Цене с клиента не доверять.
- **Order / OrderItem.** `Order`: покупатель (имя, телефон, email, комментарий/адрес), сумма, статус, `user` (FK, **nullable**), timestamps. `OrderItem`: ссылка на вариант **или** кастомная конфигурация + зафиксированная на момент покупки цена (цена «замерзает» в заказе).
- **Статусы заказа** — enum: `new → confirmed → shipped → done / cancelled`. **Стыковочная точка под оплату:** отдельное поле статуса оплаты (`pending / paid / failed`) заложить сейчас.
- **Endpoint оформления заказа.** Пересчёт суммы на сервере, проверка остатков, создание заказа, резерв/списание стока.
- **Уведомления.** Отправку в Telegram перенести на бэкенд (токен на сервере). Задел на email-подтверждение.

**Ключевые решения:** клиентская vs серверная корзина; резерв стока при заказе или списание по факту оплаты; гостевой заказ (да, обязательно).

**Готово, когда:** оформление создаёт запись в БД с пересчитанной на сервере суммой, уменьшает остатки, шлёт уведомление, поддерживает обычные и кастомные позиции.

## ✅ Этап 4. Frontend на Next.js/TypeScript

**Цель:** перенести вёрстку на React-компоненты и подключить к API вместо localStorage-заглушек.

**Задачи:**
- **Структура App Router:** `/`, `/catalog`, `/product/[slug]` (**новая страница — в статик-сайте детальной карточки нет!**), `/constructor`, `/cart`, `/checkout`, `/contacts`, `/testimonials`, `/thank-you`.
- **Типы из API.** `openapi-typescript` генерирует TS-типы из Swagger-схемы (Этап 2).
- **Рендеринг под SEO** (причина выбора Next.js): публичные страницы (каталог, карточка) — серверные компоненты / SSR или SSG. Интерактив (корзина, конструктор) — клиентские компоненты.
- **Перенос вёрстки.** CSS во многом наследуется из текущего `styles.css` (в CSS Modules / глобальные стили).
- **Состояние корзины** — контекст/стор (Zustand или Context), синхронизация с localStorage.
- **Изображения** — `next/image`.
- **Формы** — валидация оформления (имя/телефон/email), серверная валидация — источник истины.
- **SEO-обвязка:** Metadata API, `sitemap.xml`, `robots.txt`, Open Graph (динамические на карточках товара).

**Ключевые решения:** SSR vs SSG для каталога; библиотека состояния; CSS Modules vs Tailwind.

**Готово, когда:** весь путь (главная → каталог → карточка → корзина → оформление → спасибо) работает на реальном API, публичные страницы с серверным рендерингом.

# Проверка на ошибки и несоответствия между этапами

1. **Оплата отложена, но запуск от неё зависит.** MVP запускается как «приём заказов без онлайн-оплаты»; модель заказа готовится к оплате заранее (поле статуса оплаты в Этапе 3).
2. **Решение по корзине — ДО Этапов 3 и 4 одновременно.** Клиентская vs серверная влияет на бэкенд-модель и на состояние фронта. Зафиксировать в начале Этапа 3.
3. **Аутентификация на Этапе 6, но `Order.user` нужен на Этапе 3.** Поле `user` — nullable с самого начала; решение про JWT/сессии и гостевой checkout — раннее.
4. **Причина выбора Next.js (SEO) реализуется только при SSR/SSG.** На Этапе 4 публичные страницы обязательно серверные.
5. **Конструктор как товар связывает Этап 1 и Этап 3.** `OrderItem` должен уметь хранить кастомную конфигурацию, иначе оформление кастомного заказа сломается.
6. **Хранилище медиа (Этап 2) диктует деплой (Этап 7).** Выбор storage делать с оглядкой на способ деплоя бэка.
7. **Цена «замерзает» в заказе.** `OrderItem` хранит цену на момент покупки, не берёт из товара по FK. Пересчёт всегда на сервере.

**Вывод:** порядок этапов корректен. Ключевой нюанс — решения об аутентификации и типе корзины принадлежат началу (Этапы 1–3), хотя аккаунты как фича реализуются на Этапе 6.
